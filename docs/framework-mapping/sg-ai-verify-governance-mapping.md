# SecureAI Gateway - Singapore AI Verify and Model AI Governance Mapping

## Purpose of this mapping

This document helps enterprise teams discuss how SecureAI Gateway capabilities align with AI Verify-style governance testing practices and Singapore model AI governance review themes.

## Scope and disclaimer

This document is provided for enterprise evaluation and focused pilot discussions only. It does not represent certification, accreditation, approval or endorsement by IMDA, AI Verify Foundation or any regulator.

## 1. AI Usage Transparency

### Governance focus

Organisations should understand where AI is used, by whom, for which business purpose and through which model.

### How SecureAI Gateway supports this area

SecureAI Gateway is designed to support AI usage visibility by user, department, model and use case.

### Evidence available during evaluation

- Usage views
- Request-level metadata
- Model and provider selection records

### Customer responsibilities / not covered by the product

Customers define AI inventory scope, user disclosure requirements and business owner accountability.

## 2. Human Oversight and Review

### Governance focus

Higher-risk AI use cases may require review, escalation or approval workflows.

### How SecureAI Gateway supports this area

SecureAI Gateway can support policy decisions including allow, mask, block or review, with human review trigger patterns for selected scenarios.

### Evidence available during evaluation

- Policy decision records
- Trace ID
- Review-oriented audit evidence

### Customer responsibilities / not covered by the product

Customers define review thresholds, approver roles and final business decisions.

## 3. Data Governance and Sensitive-Data Protection

### Governance focus

Organisations should control sensitive data exposure when using AI systems.

### How SecureAI Gateway supports this area

SecureAI Gateway supports prompt inspection and sensitive-data masking / blocking before selected requests reach approved AI models.

### Evidence available during evaluation

- Sensitive-data handling evidence
- Masking or blocking decision examples
- Request-level audit records

### Customer responsibilities / not covered by the product

Customers define data classification, retention rules, acceptable-use policy and legal basis for processing.

## 4. Approved-Model Access and Policy Enforcement

### Governance focus

Organisations should control which users, departments and workflows may use approved AI models.

### How SecureAI Gateway supports this area

SecureAI Gateway supports approved-model access, routing controls and tenant policy enforcement.

### Evidence available during evaluation

- Approved model route settings
- Tenant policy configuration
- Routing decision audit evidence

### Customer responsibilities / not covered by the product

Customers approve model providers, regions, terms, usage scope and operating policy.

## 5. Audit Evidence and Accountability

### Governance focus

Organisations should retain evidence that supports review, investigation and accountability.

### How SecureAI Gateway supports this area

SecureAI Gateway records audit evidence with Trace ID, request context, policy decision and model-routing metadata.

### Evidence available during evaluation

- Trace ID examples
- Audit search examples
- Export or review discussion material

### Customer responsibilities / not covered by the product

Customers define retention, access control, investigation workflow and formal audit procedures.

## 6. Usage and Token Spend Visibility

### Governance focus

Organisations may need visibility into token usage and cost source status when scaling AI adoption.

### How SecureAI Gateway supports this area

SecureAI Gateway can support request-level token usage and estimated cost, with cost source marked as estimated / provider-reported where available / reconciliation-ready.

### Evidence available during evaluation

- Token usage summaries
- Estimated cost views
- Cost source status

### Customer responsibilities / not covered by the product

Customers remain responsible for financial reconciliation, provider billing records and procurement controls.

## 7. Deployment Model Considerations

### Governance focus

Organisations should select deployment models that match data, security and operational requirements.

### How SecureAI Gateway supports this area

SecureAI Gateway supports SaaS Control Plane, hybrid deployment and customer-controlled Data Plane discussion patterns.

### Evidence available during evaluation

- Deployment model notes
- Control Plane / Data Plane boundary description
- Customer-controlled Data Plane option

### Customer responsibilities / not covered by the product

Customers define network, identity, security operations, model-provider and data-location requirements.

## Summary

SecureAI Gateway is designed to support enterprise AI governance review through usage transparency, human review trigger patterns, data protection controls, approved-model access, audit evidence, token spend visibility and deployment model discussion.
