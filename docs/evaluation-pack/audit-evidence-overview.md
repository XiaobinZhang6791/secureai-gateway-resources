# SecureAI Gateway - Audit Evidence Overview

## Purpose

This document describes audit evidence that may be reviewed during SecureAI Gateway enterprise evaluation and focused pilot discussions.

## What evidence is captured

Evidence may include request metadata, Trace ID, user context, tenant context, department, use case, model route, policy decision, sensitive-data handling result and timestamps.

## Trace ID

Trace ID supports investigation and review of a governed AI request across policy, routing and audit views.

## User, tenant, department and use case visibility

SecureAI Gateway is designed to provide visibility into governed AI usage by user, tenant, department and use case.

## Model selected and routing decision

Audit evidence can include the model or provider route selected for a request and the policy basis for that route.

## Policy decision: allow, mask, block or review

Policy decisions can support review of whether a request was allowed, masked, blocked or marked for review.

## Sensitive-data handling evidence

Sensitive-data handling evidence may include matched rule type, action and masked or blocked decision metadata.

## Token usage and estimated cost source

SecureAI Gateway can support request-level token usage and estimated cost. Cost source may be marked as estimated / provider-reported where available / reconciliation-ready.

## Search, review and export considerations

Search, review and export requirements should be agreed during enterprise evaluation, including retention period and access controls.

## Review workflow during focused pilot

Focused pilot review can use sample requests, Trace ID records and policy decisions to confirm governance evidence expectations.

## Current limitations and non-claims

Audit evidence is not a final financial statement, regulator approval or substitute for customer audit procedures.

## Customer responsibilities

Customers define retention, reviewers, evidence access, escalation rules, legal hold requirements and audit operating procedures.
